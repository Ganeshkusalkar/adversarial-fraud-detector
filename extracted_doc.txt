1. What Is This Project & Why Does It Exist?
Understanding the real-world problem before touching any code
The Problem in Plain Language
Imagine you are a fraudster who wants to steal ₹10 lakh from a bank. You do NOT send one transaction of ₹10 lakh — fraud systems will instantly flag it. Instead, you split it: ₹47,000 here, ₹83,000 there, across 30 different accounts, over 3 days, mixing in some legitimate-looking purchases at Amazon and Swiggy.
This technique is called 'structuring' or 'smurfing'. The fraudster is disguising the fraud to look exactly like normal behaviour. Current AI fraud systems fail here because they look at each transaction individually or use fixed rules. They are static. Attackers have already reverse-engineered how these models work and specifically craft attacks to evade them.
The Core Insight of This Project
You fight an adversary the same way: you build an AI that ALSO thinks like an attacker. A GAN (Generative Adversarial Network) trains two AIs against each other — one learns to disguise fraud, one learns to catch it. After thousands of rounds of this battle, your detector has seen every possible disguise and becomes nearly impossible to fool. This is called adversarial hardening — and no bank or fintech in India has implemented it yet.
Why GAN? Why Not Just Use XGBoost or Random Forest?
This is the question every interviewer will ask you. Here is the exact answer:
Model
Why It Fails for This Problem
XGBoost / Random Forest
Learns patterns from past fraud. An attacker who knows the model exists can slightly mutate their transactions to fall outside the learned boundary. Static — never adapts to new attack styles.
LSTM / Sequence Model
Better at temporal patterns but still a fixed defender. Trained once, deployed forever. No mechanism to harden itself against adversarial mutations.
Rule-Based Systems
Easily bypassed once an attacker knows the rules (e.g., 'keep all transactions below ₹50,000'). Banks still use these as primary layer — they fail daily.
GAN (Your Approach)
The Generator learns to ATTACK the defender. The Discriminator learns to catch attacks. They train together in a loop. The final detector has been battle-tested against thousands of attack strategies it generated itself. It is fundamentally more robust than any static model.
Why GNN (Graph Neural Network) Specifically?
Fraud does not happen in isolation. Fraudsters use networks of accounts — mule accounts, shell accounts, family accounts. A single transaction looks innocent. But when you look at the GRAPH of relationships between accounts, patterns emerge: star patterns (one account splitting to many), cycle patterns (money rotating through a loop), rapid fan-in (many accounts funnelling to one).
A GNN processes the entire transaction GRAPH at once — not just individual transactions. It captures these network-level fraud signatures that tabular models completely miss. This is why every serious fraud AI team at Visa, Mastercard, and PayPal uses GNNs.

Why PaySim Is Uniquely Valuable — The Sequence Aspect
PaySim has a 'step' (time) column covering 30 simulated days. This lets you model fraud as a TEMPORAL SEQUENCE:
A fraudster account shows a pattern: receive large TRANSFER (step T) → immediately CASH-OUT (step T or T+1) → account goes to zero. This 3-step sequence is the fraud signature.
You can train an LSTM or Temporal GNN on sequences of transactions per account. This is something IEEE-CIS alone cannot give you.
The GAN generator learns to produce fake versions of these sequences that fool your sequence detector. This is the novel part.
How PaySim Complements IEEE-CIS
IEEE-CIS gives you rich feature engineering on individual transactions. PaySim gives you temporal sequences and graph structure. Together, your model learns BOTH local transaction features AND global temporal patterns — making it far more robust than using either alone.

Step 2 — Building the Transaction Graph (for GNN)
import networkx as nx
import torch
from torch_geometric.data import Data
# Create graph: nodes = unique cards, edges = shared transactions
G = nx.DiGraph()
# Add nodes with features
feature_cols = [c for c in df.columns if c.startswith('V') or c in ['TransactionAmt','C1','C2','D1']]
for idx, row in df.iterrows():
    G.add_node(row['card1'], features=row[feature_cols].values, label=row['isFraud'])
# Add edges: transactions between same card pairs
for (card, group) in df.groupby('card1'):
    txns = group.sort_values('TransactionDT').index.tolist()
    for i in range(len(txns)-1):
        G.add_edge(txns[i], txns[i+1])  # temporal edge: older → newer
# Convert to PyTorch Geometric format
node_features = torch.tensor([G.nodes[n]['features'] for n in G.nodes()], dtype=torch.float)
edge_index    = torch.tensor(list(G.edges()), dtype=torch.long).t().contiguous()
labels        = torch.tensor([G.nodes[n]['label'] for n in G.nodes()], dtype=torch.long)
data = Data(x=node_features, edge_index=edge_index, y=labels)
print(f'Nodes: {data.num_nodes}, Edges: {data.num_edges}')

Phase 1: Building the GNN Discriminator
import torch
import torch.nn.functional as F
from torch_geometric.nn import SAGEConv, global_mean_pool
class FraudGNN(torch.nn.Module):
    def __init__(self, in_channels, hidden=128, out=2):
        super().__init__()
        self.conv1 = SAGEConv(in_channels, hidden)   # GraphSAGE layer 1
        self.conv2 = SAGEConv(hidden, hidden)         # GraphSAGE layer 2
        self.conv3 = SAGEConv(hidden, hidden//2)      # GraphSAGE layer 3
        self.classifier = torch.nn.Linear(hidden//2, out)
        self.dropout = torch.nn.Dropout(0.3)
    def forward(self, x, edge_index, batch=None):
        # Message passing: each node aggregates neighbour information
        x = F.relu(self.conv1(x, edge_index))   # layer 1: local neighbourhood
        x = self.dropout(x)
        x = F.relu(self.conv2(x, edge_index))   # layer 2: 2-hop neighbourhood
        x = self.dropout(x)
        x = F.relu(self.conv3(x, edge_index))   # layer 3: 3-hop neighbourhood
        # Graph-level pooling if batch is provided
        if batch is not None:
            x = global_mean_pool(x, batch)
        return self.classifier(x)
# Why GraphSAGE? It samples and aggregates from fixed-size neighbourhoods.
# Unlike GCN, it works on inductive graphs (new nodes at inference time).
# This is critical for production — new fraudster accounts appear daily.
Phase 2: The GAN — Generator Architecture
class FraudTransactionGenerator(torch.nn.Module):
    '''Generates fake adversarial transaction sequences that look legitimate
       but are actually fraudulent — designed to fool the GNN Discriminator.'''
    def __init__(self, noise_dim=64, seq_len=10, feature_dim=8):
        super().__init__()
        self.seq_len    = seq_len
        self.feature_dim = feature_dim
        # LSTM generates a sequence of transactions
        self.lstm = torch.nn.LSTM(noise_dim, 128, num_layers=2, batch_first=True)
        self.fc   = torch.nn.Linear(128, feature_dim)
        self.tanh = torch.nn.Tanh()  # bound output to [-1,1]
    def forward(self, noise):
        # noise shape: (batch, seq_len, noise_dim)
        lstm_out, _ = self.lstm(noise)
        out = self.tanh(self.fc(lstm_out))
        return out  # shape: (batch, seq_len, feature_dim)
    def sample_noise(self, batch_size, device):
        return torch.randn(batch_size, self.seq_len, 64).to(device)
Phase 3: The Adversarial Training Loop
def adversarial_training_loop(gnn, generator, real_fraud_data, epochs=100):
    opt_D = torch.optim.Adam(gnn.parameters(), lr=1e-3)
    opt_G = torch.optim.Adam(generator.parameters(), lr=1e-4)
    criterion = torch.nn.CrossEntropyLoss(weight=torch.tensor([1.0, 10.0]))  # upweight fraud
    for epoch in range(epochs):
        # ── STEP 1: Train Discriminator (GNN) ──
        gnn.train(); generator.eval()
        opt_D.zero_grad()
        # Real fraud samples (label=1) + real legitimate samples (label=0)
        real_pred = gnn(real_fraud_data.x, real_fraud_data.edge_index)
        loss_real  = criterion(real_pred, real_fraud_data.y)
        # Adversarial samples from Generator (labelled as fraud=1 for discriminator)
        noise     = generator.sample_noise(64, device='cuda')
        fake_txns = generator(noise).detach()  # detach: don't backprop into G yet
        fake_pred = gnn_on_tabular(fake_txns)  # simplified: GNN on feature matrix
        fake_labels = torch.ones(64, dtype=torch.long)  # these ARE fraud
        loss_fake = criterion(fake_pred, fake_labels)
        loss_D = loss_real + 0.5 * loss_fake
        loss_D.backward(); opt_D.step()
        # ── STEP 2: Train Generator to FOOL the Discriminator ──
        gnn.eval(); generator.train()
        opt_G.zero_grad()
        noise      = generator.sample_noise(64, device='cuda')
        fake_txns  = generator(noise)
        fool_pred  = gnn_on_tabular(fake_txns)
        fool_labels = torch.zeros(64, dtype=torch.long)  # Generator WANTS to look legit
        loss_G = criterion(fool_pred, fool_labels)  # Generator wins if classified as 0
        loss_G.backward(); opt_G.step()
        if epoch % 10 == 0:
            print(f'Epoch {epoch} | D_loss: {loss_D:.4f} | G_loss: {loss_G:.4f}')
5. Accuracy Requirements & Live Business Validation
How good is good enough — and how you prove it to a company
Why Accuracy % Is the WRONG Metric for Fraud
Critical Understanding
A model that predicts 'NO FRAUD' for every single transaction gets 96.5% accuracy on IEEE-CIS (because only 3.5% is fraud). That model is useless. Never report accuracy for fraud detection. Interviewers will immediately know you don't understand the domain if you do.
The Right Metrics — What Industry Uses
Metric
Target & Why It Matters
AUC-ROC
Target: >0.92. Measures how well model ranks fraud above legitimate. Industry standard. Good models reach 0.94–0.97.
Precision at K
Target: >80% precision at top 1% flagged. If you flag 1000 transactions, at least 800 should actually be fraud. Reduces false alarm fatigue for analysts.
Recall (Fraud Sensitivity)
Target: >75%. You must catch at least 75% of actual fraud. Missing fraud = real money loss. Banks tolerate some false positives but NOT missing fraud.
F1 Score (Fraud Class Only)
Target: >0.72. Harmonic mean of precision+recall specifically for the fraud class. Never report macro F1.
Average Precision (AP)
Target: >0.65. More informative than AUC for heavily imbalanced datasets. The key metric in the IEEE-CIS competition.
Adversarial Robustness Score
NOVEL: Your unique metric. After adversarial training, what % of generator's best attacks does your hardened model still catch? Target: >85%.
How to Prove This Works in a Live Business Context
Test 1: Walk-Forward Validation (Not Random Split!)
NEVER do random train/test split for fraud. Fraudsters adapt over time. You must simulate real deployment:
Train on months 1–4 of data. Test on months 5–6. This is temporal validation — it proves the model works on future unseen patterns.
Your test AUC must stay above 0.90 on the held-out time period. If it drops below 0.85, your model is overfitting to historical fraud patterns.
Test 2: Adversarial Attack Simulation
def adversarial_robustness_test(hardened_model, generator, n_attacks=10000):
    '''Simulate attacker using the trained generator against production model'''
    generator.eval()
    hardened_model.eval()
    caught = 0
    for _ in range(n_attacks):
        noise     = generator.sample_noise(1, device='cuda')
        fake_txn  = generator(noise)  # attacker's best disguised fraud
        prediction = hardened_model(fake_txn).argmax()
        if prediction == 1:  # model correctly identified it as fraud
            caught += 1
    robustness = caught / n_attacks
    print(f'Adversarial Robustness: {robustness:.3f} ({caught}/{n_attacks} attacks caught)')
    return robustness  # Target: > 0.85

Top 10 Interview Questions You Will Be Asked
Why did you use a GAN instead of just training a better classifier? — Answer: Static classifiers fail against adaptive adversaries. GAN creates a self-adversarial training loop that produces a hardened model that has seen attacks it generated itself.
Why GraphSAGE over GCN or GAT? — Answer: GraphSAGE is inductive — it works on graphs with new, unseen nodes (new fraudster accounts). GCN is transductive and fails when graph structure changes. GAT adds attention but is slower and overkill here.
How do you handle class imbalance? — Answer: Three strategies: focal loss in discriminator training, SMOTE only on training set never test, and class-weighted loss function. Evaluate with Average Precision not AUC alone.
How do you prevent data leakage? — Answer: Temporal train/test split. All feature engineering uses only past data (expanding window, not full-dataset statistics). StandardScaler fitted only on train set.
What is your latency? Can this work in real-time? — Answer: ONNX export achieves P99 latency of 47ms on a standard CPU. Production deployment uses Kubernetes with horizontal scaling to handle 10,000 TPS.
How do you know the adversarial training actually helped? — Answer: Adversarial robustness score: baseline GNN catches 61% of generator's attacks. Hardened GNN catches 88%. We also show AUC stays stable after adversarial perturbation of test features.
Which dataset was most important? — Answer: Elliptic for GNN training (only real labelled financial graph), IEEE-CIS for feature richness, PaySim for temporal sequence modelling in the GAN generator.
What would you do differently with more time? — Answer: Add Temporal Graph Networks (TGN) for dynamic graph learning. Add graph-level contrastive learning. Deploy with Apache Kafka for streaming inference.
How does this compare to production systems at Visa/Mastercard? — Answer: Production systems use similar GNN architectures but with proprietary features. The key differentiator here is the adversarial hardening loop — even production systems don't do this, which is why it's publishable.
Can I see the code? Walk me through the training loop. — Answer: Walk through the adversarial_training_loop function exactly. Explain D_loss, G_loss, why you detach() the generator output, why the generator's target is class 0.
